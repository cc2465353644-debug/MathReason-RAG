# Baseline Analysis

## 1. 报告说明

本报告记录 Baseline 方法的实验过程和结果。Baseline 方法指不使用 RAG、CoT 和 SymPy 自验证，直接调用模型解题。

本报告会随着数据集、Prompt 和 evaluator 的更新持续完善。

---

## 2. Baseline 方法说明

Baseline 方法指在不使用 RAG 检索、不使用 CoT 专门推理模板、不使用 SymPy 自验证的情况下，直接将数学题输入本地大模型，由模型生成解题过程和最终答案。

该方法作为后续 RAG、CoT 和 RAG + CoT + SymPy 方法的对照组，用于观察不同增强方法是否能够带来效果提升。

### 2.1 模型

本实验使用的本地模型为：

```text
Qwen/Qwen2.5-1.5B-Instruct
```

该模型属于轻量级指令模型，能够在本地 GPU 环境下运行，适合作为本项目早期 Baseline 阶段的测试模型。
在 Baseline 阶段，模型只负责根据输入题目直接生成答案，不额外接收检索知识、公式提示或 SymPy 校验反馈。
模型输入为数学题文本，输出为包含解题过程和最终答案的自然语言回答。

### 2.2 Prompt

Baseline 阶段使用 Basic Prompt，主要要求模型扮演数学解题助手，并给出必要的解题过程。
Prompt 模板如下：

```text
你是一个数学解题助手。
请解答下面这道数学题，并给出必要的解题过程。

题目：{question}
```

其中 {question} 会被替换为测试集中每一道具体数学题。
该 Prompt 不包含额外知识点提示，也不强制模型按照固定步骤进行推理。因此，Baseline 结果可以反映模型在最简单提示条件下的直接解题能力。

### 2.3 解题流程

Baseline 解题流程如下：

```test
读取测试题数据
↓
提取题目内容
↓
构造 Basic Prompt
↓
调用本地模型生成回答
↓
提取模型输出
↓
保存到 baseline_results.jsonl
↓
使用 evaluator 进行初步评测
```

具体来说，系统首先从 `data/test_questions.jsonl` 中读取数学题，每道题包含题目编号、题目内容、标准答案、题型和难度等信息。
随后，系统将题目填入 Basic Prompt，并调用本地模型生成答案。模型输出结果会保存到：

```test
results/baseline_results.jsonl
```

每条结果主要包含以下信息：

```json
{
  "id": "algebra_001",
  "question": "解方程：x^2 - 5x + 6 = 0",
  "gold_answer": "x=2 或 x=3",
  "model_answer": "模型生成的答案",
  "topic": "代数方程",
  "difficulty": "easy"
}
```

最后，系统使用 evaluator 对模型答案和标准答案进行比较，得到初步评测结果。当前 evaluator 主要用于支持简单字符串匹配、数值等价判断和部分格式归一化。

### 3.7 典型误判案例

#### Case 1：多解方程表达不同

题目：

```text
解方程：x^2 - 5x + 6 = 0
```

标准答案：

```test
x=2 或 x=3
```

模型答案：

```test
x1=2, x2=3
```

初版 evaluator 判定结果：

```test
错误
```

人工复核结果：

```test
正确
```

分析：

该题模型求出的两个解为 2 和 3，与标准答案完全一致。但是由于模型输出使用了 x1=2, x2=3 的形式，而标准答案使用了 x=2 或 x=3 的形式，二者字符串不完全相同，导致初版 evaluator 将其误判为错误。
该案例说明，在数学题评测中，不能只依赖简单字符串匹配。对于方程多解问题，更合理的做法是提取答案中的数值集合，再比较两个集合是否一致。

改进方向：

```test
对方程解类题目进行答案归一化，提取所有数值解后进行集合比较。
```

#### Case 2：分数和小数表达不同

题目：

```test
一个袋子中有 3 个红球和 2 个蓝球，随机取出 1 个球，取到红球的概率是多少？
```

标准答案：

```test
3/5
```

模型答案：

```test
0.6
````

初版 evaluator 判定结果：

```test
错误
```

人工复核结果：

```test
正确
```

分析：

该题的正确概率为：

```test
3 / (3 + 2) = 3/5 = 0.6
```

模型输出的小数 `0.6` 与标准答案中的分数 `3/5` 在数值上完全等价。但是初版 evaluator 只比较字符串内容，无法判断分数和小数之间的等价关系，因此将该题误判为错误。
该案例说明，数学答案常常存在分数、小数、百分数等不同表达形式。对于数值类答案，evaluator 应该将不同表达转换为统一数值后再进行比较。

改进方向：

```test
对数值类答案加入数值等价判断，例如将 3/5 和 0.6 都转换为浮点数或 SymPy 表达式后比较。
```

### 3.8 实验结论

通过 Experiment 01，可以得到以下初步结论：

1. Baseline 解题流程已经跑通，系统能够完成读取题目、构造 Prompt、调用模型、保存输出和自动评测；
2. 在当前 10 道基础数学题上，本地模型的回答经增强 evaluator 判断均为正确；
3. 初版字符串匹配 evaluator 曾出现误判，主要原因是数学等价答案存在不同表达方式；
4. 增强 evaluator 后，当前两个典型误判问题得到解决，包括多解方程格式差异和分数/小数等价问题；
5. 当前测试集规模较小、题目难度较低，因此 100% 准确率只能说明流程和初步评测可用，不能说明模型具备稳定的复杂数学推理能力；
6. 后续需要扩展数据集，并引入更多答案类型，以进一步检验 evaluator 和模型解题能力。

本次实验的主要价值不在于证明模型能力很强，而在于建立了后续实验所需的基础评测流程。后续的 RAG、CoT 和 SymPy 自验证方法都将基于该 Baseline 结果进行对比

## 4. Evaluator 当前局限

当前 evaluator 仍然是初步版本，主要用于完成 Baseline 阶段的基础评测。虽然它已经能够处理部分简单答案格式差异，但仍然存在明显局限。

### 4.1 对答案格式较敏感
数学答案可能存在多种正确表达方式，例如：

```text
x=2 或 x=3
x1=2, x2=3
{2, 3}
2 和 3
```

这些表达在数学意义上等价，但字符串形式不同。当前 evaluator 只能处理部分简单格式，面对更复杂的表达仍可能误判。

### 4.2 对复杂表达式支持不足

当前 evaluator 对简单数值和部分格式归一化有一定支持，但对复杂代数表达式、三角恒等式、导数结果、积分结果等还不够稳定。
例如：

```text
2x + 2
2(x + 1)
```
这两个表达式数学等价，但如果只依赖字符串匹配，仍然可能被判断为不同。
后续需要引入 SymPy，对表达式进行化简和等价判断。

### 4.3 对区间类答案支持不足

函数单调区间、不等式解集、定义域和值域等题目经常会出现区间形式答案，例如：
```text
(-∞, 1)
[0, 2]
x > 3
```

这类答案的格式变化较多，当前 evaluator 还不能稳定解析和比较。

### 4.4 对文字解释类答案无法自动判断
对于概念解释题、证明题或需要文字说明的题目，答案不一定是固定数值或表达式。
例如：

```text
为什么函数在该区间上单调递增？
```

这类题目需要判断推理过程是否合理，而不是只判断最终答案。当前 evaluator 暂时无法处理这类开放式答案，只能依赖人工检查。

### 4.5 对模型输出的最终答案提取能力有限
模型输出通常包含完整解题过程，而最终答案可能出现在最后一行，也可能夹杂在中间步骤中。
例如：

```text
因此，方程的解为 x=2 或 x=3。
```

当前 evaluator 对最终答案的提取还比较简单。如果模型输出较长、格式不统一，就可能影响判断结果。

### 4.6 当前 evaluator 的定位
因此，当前 evaluator 更适合作为 Baseline 阶段的初步自动评测工具，而不是最终版本的数学判题系统。
后续优化方向不是针对单个题目写特殊规则，而是按照答案类型逐步增强，例如：
| 答案类型    | 改进方向                 |
| ------- | -------------------- |
| 数值类答案   | 支持整数、分数、小数、百分数等价判断   |
| 表达式类答案  | 使用 SymPy 判断表达式是否等价   |
| 方程解类答案  | 提取解集并进行集合比较          |
| 区间类答案   | 解析区间边界和开闭情况          |
| 选择题答案   | 统一选项格式               |
| 文字解释类答案 | 暂时人工复核，后续考虑 LLM 辅助评审 |

## 5. 后续实验计划

后续实验将围绕 Baseline 结果继续推进，重点是扩展数据集、增强 evaluator，并逐步引入 RAG、CoT 和 SymPy 自验证方法。

### 5.1 扩展测试集

当前 `test_questions_v0.1` 只有 10 道基础题，主要用于验证流程。后续计划扩展到 30–50 道题，覆盖更多数学题型。

计划增加的题型包括：

```test
一元二次方程
代数化简
函数求导
简单积分
极限计算
数列求和
概率计算
几何计算
三角函数
函数单调性
不等式求解
```

同时，为每道题增加更明确的字段，例如：

```json
{
  "id": "algebra_001",
  "question": "解方程：x^2 - 5x + 6 = 0",
  "answer": "x=2 或 x=3",
  "topic": "代数方程",
  "difficulty": "easy",
  "answer_type": "equation_roots"
}
```

其中 `answer_type ` 用于指导 evaluator 采用不同的评测方法。

### 5.2 增强 evaluator

后续 evaluator 将从简单字符串匹配逐步升级为按答案类型判断。

计划支持：
1. 数值等价判断；
2. 分数、小数、百分数统一转换；
3. 方程多解集合比较；
4. 简单代数表达式等价判断；
5. 使用 SymPy 判断化简、导数、积分结果；
6. 对无法自动判断的题目标记为 `need_review` ，避免强行误判。

目标不是让 evaluator 一次性完美，而是让它能够逐步覆盖更多数学答案类型。

### 5.3 引入 CoT Prompt 对比实验

在 Baseline 方法之后，将设计 CoT Prompt，引导模型按照更明确的步骤进行解题。

对比内容包括：

```test
Baseline Prompt
CoT Prompt
```

重点观察：
1. CoT 是否能提升答案正确率；
2. CoT 是否能让解题过程更清晰；
3. CoT 是否会增加冗余输出；
4. CoT 是否会影响 evaluator 提取最终答案。

### 5.4 构建 RAG 知识库

后续将整理数学知识库，包括常用公式、知识点说明和典型例题。

RAG 阶段目标是实现：

```test
用户题目
↓
检索相关知识点
↓
将知识点加入 Prompt
↓
模型生成答案
```

重点观察：
1. RAG 是否能减少公式选择错误；
2. RAG 是否能提高中等难度题目的解题稳定性；
3. 检索结果是否真正与题目相关；
4. 不相关检索内容是否会干扰模型回答。

### 5.5 引入 SymPy 自验证

在后续阶段，将使用 SymPy 对部分数学计算进行自动验证。
优先验证的内容包括：

```test
方程求解
代数化简
函数求导
简单积分
数值计算
```

目标是让系统不只依赖模型生成答案，而是能够通过工具检查部分关键计算，从而减少计算错误。

### 5.6 后续方法对比

最终计划对比以下几种方法：

| 方法                | 是否使用 RAG | 是否使用 CoT | 是否使用 SymPy |
| ----------------- | -------- | -------- | ---------- |
| Baseline          | 否        | 否        | 否          |
| CoT               | 否        | 是        | 否          |
| RAG               | 是        | 否        | 否          |
| RAG + CoT         | 是        | 是        | 否          |
| RAG + CoT + SymPy | 是        | 是        | 是          |


后续将通过准确率、自动判定率、错误类型分布和典型案例分析，比较不同方法的效果差异。
